# 🤖 AI 部署纪实：在 Android 荒野上的生存指南

## 📖 前言
这份文档记录了我们在将 `phone-vision-memory` 部署到安卓端侧（特别是 ZA2232T6XT 设备）过程中，那些令 AI 程序员感到“秃头”的技术苦衷。这不仅仅是日志，更是一份在极端受限环境下进行自动化部署的经验沉淀。

---

## 🛠️ 技术苦衷与“避坑”指南

### 1. 嵌套指令的“转义地狱” (The Escaping Hell)
*   **现象**：我们的操作链条极长：`本地 PC -> Windows 跳板机 -> ADB 隧道 -> Android Shell -> run-as com.termux -> Python`。
*   **苦衷**：每一层 Shell 都会吞掉一部分引号或反斜杠。为了在最底层的 Python 里运行一个简单的字符串，我必须在最外层写出像 `\\\\\\\"` 这样丧心病狂的转义符。
*   **结论**：**不要在命令行里肉搏。** 以后优先将复杂逻辑写成本地 `.py` 文件，通过 `adb push` 整体分发。

### 2. 非 Root 环境下的“隐形枷锁” (The Non-Root Cage)
*   **现象**：由于设备没有 Root 权限，我们被迫使用 `run-as com.termux` 这种“借用权限”的方式。
*   **苦衷**：
    *   **权限孤岛**：即使是 ADB Root 也进不去 Termux 的家目录。
    *   **环境贫瘠**：`run-as` 启动的 Shell 几乎没有任何环境变量（甚至连 `PATH` 都没有）。
    *   **内核壁垒**：我无法执行 `setenforce 0` 来临时放宽安全限制，遇到 SELinux 拦截时只能束手就擒。

### 3. 跨系统传输的“二进制污染” (CRLF Pollution)
*   **现象**：通过 Windows 跳板机转发文件时，Python 库莫名报语法错误。
*   **苦衷**：Windows 系统的 `\r\n`（回车换行）会像病毒一样混入二进制流。一旦库文件被污染，Python 就会报出极具误导性的 `SyntaxError`。
*   **结论**：在没有稳定二进制传输通道时，**使用 Base64 进行单行编码传输** 是最后的尊严，或者干脆在目标机器上重新 `pip` 下载。

### 4. Android 系统的“关禁闭”策略 (App Freezing)
*   **现象**：网络明明是通的（Root 身份），但 Termux 访问网络报错 `Operation not permitted`。
*   **真凶**：Android 11+ 为了省电，会封锁长时间不在前台运行的 App 的网络 Socket。
*   **神药**：**手动点开一次 Termux App。** 这道“人肉特赦令”比任何 ADB 指令都管用。

### 5. 端口映射的“迷魂阵” (Port Mapping Chaos)
*   **现象**：`localhost:8080` 在手机上能通，但在 Win 机器上不通。
*   **苦衷**：由于多台手机共用一个 Win 跳板机，端口冲突频发。必须精准管理 `adb forward` (PC 连手机) 和 `adb reverse` (手机连 PC) 的映射关系。

## 💡 给未来的自己
*   **尊重设备**：如果设备是你的“身体”，那么 Termux 就是它的“大脑”。在大脑还没醒（前台运行）之前，不要指望它能指挥身体（联网）。
*   **保持纯净**：网络通畅时，永远优先使用官方 Repo 安装（`pkg install`），不要尝试手动搬运库文件。
*   **记录就是力量**：如果你不记录这些苦衷，下一次部署时你还会认为自己是个“蠢货”。
